# Input Validation

The pipeline's attack surface is the ADO work item description. All external input
enters through that single channel and is validated before any code is generated.

## Validation Layers

### Layer 1 — ADO trigger filter (Orchestrator)

The Orchestrator only processes work items that satisfy all three conditions:
1. Type is `Feature` or `User Story`
2. State is `New`
3. Tagged with `ai-pipeline-trigger`

Work items that do not meet these criteria are silently skipped. This prevents
accidental or malicious work items from entering the pipeline.

### Layer 2 — Clarification Agent scoring

Every work item is scored 0–100 by the Clarification Agent before any downstream agent
runs. The score reflects how actionable and unambiguous the requirement is.

| Score | Action |
|---|---|
| ≥ 80 | Proceed with full pipeline |
| 50 – 79 | Post clarifying questions to ADO; block until score reaches 80 |
| < 50 | Hard block — post questions to ADO, set state to `Needs Info`, halt |

This ensures vague, incomplete, or potentially adversarial requirements are never
passed to code-writing agents.

### Layer 3 — Pydantic contract validation (inter-agent)

Every inter-agent handoff uses a Pydantic-validated JSON schema defined in
`pipeline/contracts/`. The Orchestrator validates each agent's output against its
schema before passing it downstream. A schema validation failure halts the pipeline
and triggers the diagnostic retry flow.

Key contracts and what they validate:

| Contract | File | Validates |
|---|---|---|
| `StructuredSpec` | `contracts/structured_spec.py` | Spec fields, score, acceptance criteria |
| `LLDDocument` | `contracts/lld_document.py` | File paths, endpoint shapes, required fields |
| `ChangeSummary` | `contracts/change_summary.py` | Files modified, self-review result, branch name |
| `AuditReport` | `contracts/audit_report.py` | Category scores, blocking findings, composite score |
| `TestResults` | `contracts/test_results.py` | Pass/fail counts, coverage %, per-file breakdown |

### Layer 4 — File path boundary guards

Frontend and Backend agents enforce a path prefix check on every file they write.
Any path outside the agent's designated directory raises `RuntimeError` before the
file is touched. This prevents prompt injection in a work item description from
causing code to be written to unexpected locations (e.g. pipeline source files).

## Prompt Injection Risk

Work item descriptions are passed verbatim to the Clarification Agent's Claude call.
A malicious description could attempt to override the agent's instructions. Mitigations:

- The Clarification Agent's system prompt is loaded from a file at runtime
  (`pipeline/prompts/clarification.md`) and is not influenced by the work item content
- The work item description is passed as user-turn content, not injected into the
  system prompt
- Pydantic validation at Layer 3 rejects any output that does not conform to the
  expected schema, even if an injected instruction produced unexpected JSON keys
- The score gate at Layer 2 means a work item that fails to produce a coherent spec
  never advances past clarification

## What Is Not Validated

- The content of Gherkin scenarios generated by the Story Writer (trusted as Claude output)
- The prose inside ADO comments posted by agents (not executable)
- Git commit messages (not parsed by downstream agents)
